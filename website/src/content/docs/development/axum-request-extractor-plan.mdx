---
title: "Axum Request Extractor Plan"
description: "Design notes for expanding unary rust-axum request data into explicit parameters."
---

## Context

The current `rust-axum` generator still wraps unary request data in
`xidl_rust_axum::Request<T>`, even though source-specific values such as path,
query, header, cookie, and body parameters are already modeled explicitly.

## Goal

For unary HTTP handlers, generated business traits should move from:

```rust
async fn op(&self, req: xidl_rust_axum::Request<OpRequest>) -> Result<T>;
```

to a shape closer to:

```rust
async fn op(&self, a: A, b: B, c: C) -> Result<T>;
```

## Why

- business traits should expose domain inputs, not transport wrappers
- headers are now first-class IDL parameters
- the generated API becomes easier to read and implement

## Proposed approach

1. keep `http_hir` source metadata as-is
2. render unary trait signatures from expanded parameters
3. keep helper structs only for adapter extraction when still useful
4. let the server adapter extract transport data, then call `svc.method(...)`
5. keep response modeling unchanged for now

## Scope

This plan only targets unary handlers first. Streaming operations can stay on
the current generated shape until a separate design pass.
