# HTTP

XIDL 的 HTTP 家族不是单一功能，而是一组彼此关联的能力：

1. 普通 HTTP 映射
2. HTTP Stream 映射
3. HTTP 安全映射

最常见的输出目标是：

- `rust-axum`
- `openapi`

## 最基本的 HTTP 映射

对方法使用 HTTP 动词注解：

```txt
interface HelloWorld {
    @post(path = "/hello")
    void sayHello(in string name);
};
```

如果你不写 HTTP 动词，默认等价于 `@post`。

## 参数来源规则

最常见的参数来源注解有：

- `@path`
- `@query`
- `@body`
- `@header`
- `@cookie`

实践上记住这些规则就够了：

- 路径变量用 `@path`
- 查询字符串用 `@query`
- `POST` / `PUT` / `PATCH` 的未标注输入通常进入请求体
- `GET` / `DELETE` / `HEAD` / `OPTIONS` 的未标注输入通常进入查询字符串

一个重要补充：

- 如果参数名已经出现在 route template 里，例如 `/users/{id}`，那它会被推断为 `@path`
- 如果参数名已经出现在 query template 里，例如 `{?lang}`，那它会被推断为 `@query`

### 推荐心智模型

写 HTTP IDL 时，建议按这个顺序思考：

1. 先定 HTTP 方法和 route
2. 再把路径参数标成 `@path`
3. 再决定哪些参数明确进 query / body / header / cookie
4. 只有在“默认推断刚好就是你想要的语义”时，才省略参数来源注解

这样生成结果最稳定，也最容易读。

## 一个典型例子

```txt
interface UserApi {
    @get(path = "/users/{id}{?lang}")
    string getUser(
        @path("id") string id,
        @query("lang") @optional string locale,
        out string display_name
    );

    @post(path = "/users")
    string createUser(
        string display_name,
        string email,
        out string user_id
    );
};
```

上面这个例子同时覆盖了几件事：

- `id` 从路径读取
- `lang` 是可省略查询参数
- 返回值和 `out` 参数会一起进入响应体

## route template 和多路由

XIDL 的 HTTP route 不只是一个纯字符串。常用形式有：

- 普通路径变量：`/users/{id}`
- catch-all 路径：`/files/{*path}`
- query template：`/users/{id}{?lang,region}`
- 一个方法绑定多个路由：多个 `@path(...)` 或多个带 `path = ...` 的 HTTP 动词注解

例如：

```txt
interface UserApi {
    @get(path = "/v2/users/{id}")
    @path("/users/{id}")
    @path("/u/{id}")
    string getUser(
        @path("id") string user_id,
        @query("lang") @optional string locale
    );
};
```

实践上建议：

- 对外兼容旧路径时，用多路由而不是直接改原路径
- 路径变量名和 `@path("...")` 名称保持一致
- catch-all 场景尽量放在 route 末尾

## 请求体 shaping

请求体最常见有三种形状：

1. 单个 body 参数，默认按参数名包一层对象
2. 单个 body 参数，使用 `@flatten` 直接展开
3. 多个 body 参数，按字段名组合成对象

例如：

```txt
struct CreateUserRequest {
    string name;
    @optional string alias;
    sequence<string> tags;
};

interface UserApi {
    @post(path = "/users")
    string createUser(CreateUserRequest req);

    @post(path = "/users/flat")
    string createUserFlat(@flatten CreateUserRequest body);

    @post(path = "/users/search")
    string searchUser(string keyword, uint32 page);
};
```

经验上：

- 业务上本来就有“请求对象”时，优先传一个 struct
- 只有 wire format 必须直接展开时，再用 `@flatten`
- 如果方法参数名叫 `req`、`request` 之类，生成器链路里可能更容易和内部变量混淆；示例里更推荐用明确的业务名，例如 `body`、`payload`、`resource_body`

## 生成结果

`rust-axum` 当前通常会生成：

- 业务逻辑 trait
- 暴露路由的服务包装器
- 客户端包装器
- 传输辅助类型

`openapi` 当前通常会生成：

- `openapi.json`

如果你需要文档或下游生态集成，通常会把 `rust-axum` 和 `openapi` 一起生成。

## media type 继承和覆盖

`@Consumes` 和 `@Produces` 都支持接口级默认值，以及方法级覆盖。

```txt
@Consumes("application/json")
@Produces("application/json")
interface ScopeApi {
    @post(path = "/v1/default")
    string defaultScope(string name);

    @post(path = "/v1/form")
    @Consumes("application/x-www-form-urlencoded")
    string submitForm(string name, uint32 age);

    @get(path = "/v1/msgpack/{id}")
    @Produces("application/msgpack")
    string getPacked(@path("id") string id);
};
```

建议把它理解成：

- 接口级注解是“默认配置”
- 方法级注解是“局部覆盖”

如果你的接口大多数操作都用同一种请求/响应媒体类型，接口级默认值通常更干净。

## deprecated 的三种常用写法

HTTP 映射里常见的废弃声明有：

- `@deprecated`
- `@deprecated("2026-01-01")`
- `@deprecated(since = "...", after = "...")`

它们可以放在：

- 接口上
- 方法上
- attribute 上

实践上推荐：

- 想表达“已经不建议使用”，但没有具体时间点：用 `@deprecated`
- 想表达“从某天开始废弃”：用 `@deprecated("...")`
- 想同时表达废弃时间和计划移除窗口：用 `@deprecated(since = "...", after = "...")`

接口级 `@deprecated` 会被其中的方法和 attribute 继承；方法级再声明时，以更具体的方法级为准。

## HTTP Stream

HTTP Stream 在普通 HTTP 映射上扩展长连接流式交互。

当前常见注解：

- `@server_stream`
- `@client_stream`
- `@bidi_stream`
- `@stream_codec("ndjson" | "sse")`

实践上：

- 服务器推流最常见
- `ndjson` 是默认流编码
- `sse` 主要用于服务端流
- 这部分能力存在，但比普通 HTTP 映射更偏实现驱动

## HTTP 安全

安全模型把认证声明附着在接口或方法上，而不是塞进业务参数列表。

常见注解：

- `@no_security`
- `@http_basic`
- `@http_bearer`
- `@api_key(...)`
- `@oauth2(...)`

通常规则：

- 接口级声明是默认值
- 方法级声明会替换继承来的默认值
- `@no_security` 会显式清空继承的安全要求

一个常见组合：

```txt
@http_basic(realm = "admin")
@api_key(in = "query", name = "org_key")
interface AdminApi {
    @get(path = "/reports/{id}")
    string getReport(@path("id") string id);

    @http_bearer
    @api_key(in = "cookie", name = "session_key")
    @post(path = "/reports/search")
    string searchReport(string keyword);

    @no_security
    @get(path = "/health")
    string health();
};
```

这里的含义是：

- `getReport` 继承接口级安全要求
- `searchReport` 用自己的方法级安全要求替换接口级默认值
- `health` 显式公开访问

如果你的 API 有“绝大多数私有，少数公开”的结构，接口级默认 + 方法级 `@no_security` 往往是最省事的写法。

## `@optional` 在 HTTP 里的重要性

HTTP 里“参数缺席”和“参数值为空”不是一回事。

所以：

- 可选查询参数可以完全省略
- 可选请求体字段可以不出现
- 路径参数不能是可选的

如果你关心“调用方没有传”这个事实本身，而不只是零值，那就应该显式写 `@optional`。

## 一组实用建议

- 优先把复杂输入建模成 struct，而不是堆很多松散参数
- 对外兼容旧路径时，用多 route 过渡
- route、参数来源、media type、安全要求，尽量分层表达，不要把所有语义都挤进一个方法签名里
- 想依赖默认规则时，先确认这个默认是否真的和 HTTP 方法匹配
- 需要下游文档或网关时，尽早一起生成 `openapi`

## 推荐工作流

1. 用 IDL 写出接口和 HTTP 注解
2. 生成 `rust-axum`
3. 同时生成 `openapi`
4. 需要精确规则时再看 RFC
