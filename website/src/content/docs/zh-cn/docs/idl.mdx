# IDL 语言

XIDL 建立在 OMG IDL 之上，但它不是纯粹的“语法兼容层”。在这个仓库里，
IDL 还是生成器、协议映射和工具链的输入格式。

## XIDL 在标准 IDL 之上加了什么

- HTTP 和 JSON-RPC 相关注解
- Rust 等目标的透传注解
- `#pragma xidlc ...` 形式的编译器指令

## 常见声明

当前仓库主要围绕这些声明工作：

- `module`
- `interface`
- `struct`
- `union`
- `enum`
- `bitmask`
- `bitset`
- `typedef`
- `const`
- `exception`
- `attribute`
- `native`

## 基础类型

常用标量包括：

- 整数：`short`、`long`、`long long`、`int32`、`uint64`
- 浮点：`float`、`double`、`long double`
- `boolean`
- `char`、`wchar`
- `string`、`wstring`
- `octet`
- `fixed`
- `any`、`object`、`valuebase`

## 复合类型

### `sequence<T>`

用于表示有序重复值，也是流式接口里最重要的集合类型之一。

### `map<K, V>`

用于表示键值映射，具体支持程度取决于生成目标。

### 数组

用于表达固定维度的数据。

### 构造类型嵌套

你可以在 `struct`、`union` 等声明里嵌套使用其他类型。

## 一个最小示例

```txt
module city {
    struct Citizen {
        string id;
        string display_name;
        @optional string phone_number;
    };

    interface CitizenApi {
        Citizen getCitizen(in string id);
    };
};
```

## `interface`、参数方向和返回值

接口定义方法，方法再映射到 HTTP、JSON-RPC 或其他目标。

方向规则：

- `in`：只出现在请求侧
- `out`：只出现在响应侧
- `inout`：请求和响应两边都参与

这些规则对 HTTP 和 JSON-RPC 尤其重要，因为它们直接决定：

- 哪些参数进请求体
- 哪些参数变成结果对象
- 哪些参数需要单独的包装类型

## `@optional`

`@optional` 是最常用也最重要的注解之一。

它的意义不是“可空”这么简单，而是“允许省略”。

在当前仓库里，它通常会影响：

- Rust 里的 `Option<T>`
- HTTP / OpenAPI 的可选字段与必填字段
- TypeScript 的可选属性

实践建议：

- 在“缺省”和“有值”语义不同的地方使用它
- 不要把它用于 HTTP 路径参数

## `attribute`

IDL attribute 会生成隐含的访问操作。

```txt
interface ConfigApi {
    readonly attribute string version;
    attribute boolean maintenance_mode;
};
```

不同目标可能把它映射成：

- getter / setter
- Schema 条目
- 某些流式配置接口

## 注解和 pragma

注解控制映射和生成行为，pragma 提供编译器级配置。

示例：

```txt
@rust-serde(rename_all = "camelCase")
struct User {
    @rust-serde(rename = "userId")
    string user_id;
};
```

```txt
#pragma xidlc package Smart City Public APIs
#pragma xidlc version v2.0.0
```

## 语法如何影响生成结果

- 数据声明决定目标语言中的类型
- 接口决定服务、客户端或 Schema 的结构
- 协议注解决定 HTTP / JSON-RPC 的映射方式
- 目标透传注解决定生成代码上的额外属性
