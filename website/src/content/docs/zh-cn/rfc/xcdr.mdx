# XCDR 序列化说明

本页整理 XIDL 里与 XCDR 相关的主要序列化形式，帮助你理解
`#pragma xidlc serialize(...)` 等配置的实际含义。

## XCDR1 / XCDR2 概览

你可以先把它理解成两代编码规则：

- `XCDR1`：更传统、对齐更严格
- `XCDR2`：更新、在很多场景下更紧凑

## 常见序列化形式

### `PLAIN_CDR`

- 属于 XCDR1
- 适用于 `FINAL` 和 `APPENDABLE`
- 按声明顺序序列化
- 对齐遵循自然边界

### `PL_CDR`

- 属于 XCDR1
- 主要用于 `MUTABLE`
- 通过参数列表携带成员 ID 和长度
- 适合需要更灵活演进的类型

### `PLAIN_CDR2`

- 属于 XCDR2
- 适用于 `FINAL`
- 相比 XCDR1 放宽了一些对齐要求
- 对可选成员的表示方式更现代

### `DELIMITED_CDR`

- 属于 XCDR2
- 适用于 `APPENDABLE`
- 在对象体前加长度头，便于跳过未知追加字段

### `PL_CDR2`

- 属于 XCDR2
- 主要面向 `MUTABLE`
- 结合参数化布局和较新的编码规则

## 为什么你可能需要关心它

在很多纯 HTTP 或 JSON-RPC 场景里，你可能暂时感受不到 XCDR 的存在。

但一旦你需要：

- 更稳定的二进制兼容性
- 更明确的类型演进规则
- 面向 DDS / 更底层序列化场景的控制

XCDR 选择就会变得重要。

## 在 XIDL 里怎么表达

你可以用 pragma 明确指定：

```txt
#pragma xidlc XCDR1
#pragma xidlc XCDR2
```

或者更具体：

```txt
#pragma xidlc serialize(PLAIN_CDR2)
```

## 一个实用建议

如果你现在主要在使用：

- `rust-axum`
- `rust-jsonrpc`
- `openapi`
- `openrpc`

那你通常可以先把 XCDR 视为“高级控制项”，只有在需要处理二进制兼容性、
底层序列化布局或相关目标时再深入。
