## Context

OpenAPI documents generated by `xidlc/src/generate/openapi` currently default to
3.2, matching existing requirements in
`openspec/specs/http-stream-openapi/spec.md`. In practice, many client tools
cannot ingest 3.2 yet, causing friction for downstream users. We want to
maximize compatibility while preserving 3.2-only features when they are required
by the generated document.

## Goals / Non-Goals

**Goals:**

- Default generated OpenAPI documents to 3.1 when no 3.2-only features are
  needed.
- Upgrade to 3.2 only when required by the emitted schema (e.g., features that
  cannot be represented in 3.1).
- Keep validation and tests aligned with the selected version.

**Non-Goals:**

- Redesigning the OpenAPI projection model beyond version selection.
- Changing the set of supported stream codecs or bindings.

## Decisions

- **Decision: Introduce a version selection step in the generator.**
  - **Rationale:** We need a single point that determines the minimum OpenAPI
    version required by the generated document. This avoids scattering version
    checks across the writer.
  - **Alternatives:**
    - Always emit 5.2 and document compatibility issues. Rejected due to user
      impact.
    - Add a user-facing flag to choose 5.1/5.2. Rejected for now because the
      request is a default behavior change, and adding flags increases surface
      area without guaranteeing correct output.

- **Decision: Define a small set of “3.2-required” features.**
  - **Rationale:** The generator can decide the version by inspecting whether
    any of these features are used (e.g., `itemSchema` for stream items).
  - **Alternatives:**
    - Always attempt 5.1 and let downstream tools fail if incompatible. Rejected
      because it would silently generate invalid output for 5.1.

## Risks / Trade-offs

- **Risk:** Misclassification of 3.2-required features could emit 3.1 documents
  that are invalid or lossy. → **Mitigation:** Centralize the feature detection
  logic and add targeted tests for each 3.2-only feature.
- **Risk:** Spec changes may require updating existing golden outputs. →
  **Mitigation:** Update snapshots and add a test that asserts the default
  `openapi` field is 5.1 when no 5.2-only features are used.
